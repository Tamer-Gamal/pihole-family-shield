The idea

i. How a little box blocks ads everywhere

Every time a device opens a website, it first asks a phone book of the internet - called DNS - "what's the address for this name?". Ads and trackers come from their own names (like ads.example.com). Pi-hole becomes your home's phone book, and simply refuses to look up the bad names. If the address is never found, the ad never loads.

Why it protects the whole house: because every device already uses DNS, you only change it in one place - your router - and every gadget is covered at once.
Family safety · the main reason to do this

♥ What your family is protected from

This kit is built for families first. When you run the installer with family mode on (it's the default), the same DNS trick that blocks ads is used to keep the whole network kid-safer - automatically, on every device.

🔍 SafeSearch, forced

Google, Bing and DuckDuckGo are quietly locked into SafeSearch, so adult results and images don't show up - even if a child turns SafeSearch off in the browser.

▶️ YouTube restricted

YouTube is put into Restricted Mode, which hides mature videos and comments across the network.

🚫 Adult sites blocked

Known pornographic and adult websites are blocked for every device, using actively-maintained family blocklists.

🎰 Gambling & scams

Gambling/betting sites and known scam, phishing and malware domains are blocked too - protecting wallets as well as eyes.

Please read - no filter is perfect. DNS filtering blocks known bad websites and forces safe search, but it cannot see inside a site you allow (that's what Layer 2 · HaramBlur below is for), cannot catch a brand-new adult site until the lists update, and a determined, tech-savvy child can bypass it with a VPN. Treat this as a strong first layer - use it together with device parental controls (Apple Screen Time, Google Family Link) and honest conversations. It is not a substitute for supervision.

🧩 Layer 2 · HaramBlur - blur what Pi-hole can't see inside

Pi-hole guards the network, so it can't reach inside a page it allows - and it can't install HaramBlur for you either. HaramBlur is a browser add-on: its AI runs on each device, doing the job the network physically can't. It's a free, open-source extension that automatically blurs inappropriate images and videos right on the page. Install it in each family member's browser and it works right away, no Pi-hole changes needed.

Install (per browser)

Chrome / Edge / Brave / Firefox (also Firefox for Android). Then set the blur strength & strictness in its pop-up. All detection runs in the browser - nothing is uploaded.

Lock it on (so kids can't remove it)

You can't push it from the Pi, but on a device you manage you can force-install it with a browser policy so it can't be turned off. Copy-paste steps for Chrome/Edge and Firefox (with the verified IDs) are in companions/haramblur.md.

Two layers, one goal. Pi-hole stops bad sites for every device on the network; HaramBlur blurs bad images on the sites that remain. Together they cover far more than either alone. HaramBlur is a separate project (github.com/alganzory/HaramBlur), not part of this kit - install it per browser; it needs no Pi-hole changes.
Make it harder to bypass (optional, for older kids)

In setup.conf set BLOCK_DNS_BYPASS="true" to also block the VPN / "private DNS" tricks kids use to escape the filter - then, in your router, force all devices to use only this Pi for DNS. Heads-up: this can also block a VPN you use for work, so turn it on only if you understand that trade-off.

Step 1

1. Gather what you need

Tick these off. Any recent Raspberry Pi works - a Pi 4 is the sweet spot, but a Pi 3, Pi 5, or even a Pi Zero 2 W will happily block ads for a busy household.

  • A Raspberry Pi - Pi 3 / 4 / 5 or Zero 2 W, with its power supply.
  • A microSD card - 8 GB or larger.
  • An SD card reader for your computer.
  • A network connection for the Pi - an Ethernet cable is easiest; Wi-Fi also works.
  • Access to your router's settings - the sticker on it usually has the web address, username & password.
Tip: plug the Pi in with an Ethernet cable if you can. It's more reliable for an always-on box, and it skips the Wi-Fi setup entirely.
Step 2

2. Put the operating system on the card

You'll use the free, official Raspberry Pi Imager app to write "Raspberry Pi OS Lite" onto the microSD card. "Lite" has no desktop - perfect for a box that just quietly runs.

  • Install Raspberry Pi Imager from raspberrypi.com/software.
  • Choose OS -> Raspberry Pi OS Lite (64-bit). Choose your microSD as the storage.
  • Click Edit Settings and set: hostname pihole, a username + password, your Wi-Fi (if not using Ethernet), and your timezone.
  • In the Services tab, turn on SSH (password authentication). This lets you control the Pi with no screen attached.
  • Write the card, then put it in the Pi, connect the network, and power it on. Wait ~90 seconds.
Double-check the storage you pick in the Imager - writing the card erases everything on it. Make sure it's the SD card, not a USB drive you care about.

Full picture-by-picture version: pi-hole/scripts/flash-and-first-boot.md

Step 3

3. Connect to the Pi from your computer

Open a terminal - on Windows that's PowerShell or Windows Terminal; on Mac it's Terminal. Type this (use the username you chose), and enter the password when asked:

$ ssh pi@pihole.local
If pihole.local doesn't work: open your router's device list, find the Pi's IP address (like 192.168.1.42), and use ssh pi@192.168.1.42 instead. Say yes the first time it asks about a fingerprint.
  • You see a prompt ending in pihole:~ $ - you're inside the Pi. 🎉
Step 4

4. Run the installer

Get the kit onto the Pi, set your password in one small file, and run one command; the installer does the rest (updates, Pi-hole, family filtering, blocklists, and optionally private DNS).

On the Pi, download the kit and open the settings file:

$ sudo apt-get update && sudo apt-get install -y git
$ git clone https://github.com/Tamer-Gamal/pihole-family-shield.git
$ cd pihole-family-shield/scripts
$ cp setup.conf.example setup.conf
$ nano setup.conf
Prefer offline / no repo? Copy the kit folder from your computer instead: scp -r ./pi-hole pi@pihole.local:~/ (or a USB stick), then cd ~/pi-hole/scripts. Any method that lands the kit on the Pi works.

In setup.conf, change one thing you must: the admin password. Save with Ctrl+O, Enter, then exit with Ctrl+X. (Want maximum privacy? Set INSTALL_UNBOUND="true" too.) Then run the installer:

$ sudo ./bootstrap.sh
Where does Pi-hole actually come from? This kit ships no copy of Pi-hole. Under the hood, bootstrap.sh downloads and runs Pi-hole's official installer straight from the real project - install.pi-hole.net (source: github.com/pi-hole/pi-hole, website pi-hole.net). So you always get their genuine, security-patched release; our scripts only add the family filtering and setup on top.
  • Changed PIHOLE_ADMIN_PASSWORD in setup.conf.
  • Ran sudo ./bootstrap.sh and it finished with "Pi-hole is ready!" and an admin address.
  • Wrote down the admin page address it printed (like http://192.168.1.42/admin).
It's safe to re-run. If anything hiccups (a dropped connection, a slow mirror), just run sudo ./bootstrap.sh again - it picks up where it left off.
Step 5 · the important one

5. Send your whole network through Pi-hole

Right now only the Pi knows about itself. This step tells your router to hand out the Pi as everyone's phone book - so every device is protected automatically.

Do this (recommended)

In your router's settings, find DHCP or LAN settings and set the DNS server to your Pi's IP address (the one from Step 4). Save, then reboot your devices (or turn Wi-Fi off & on).

Also do this

While you're in the router, make a DHCP reservation for the Pi so its address never changes. Look for "Reserve IP", "Static lease", or "Address reservation", and pin the Pi to its current IP.

  • Set the router's DNS to the Pi's IP address.
  • Made a DHCP reservation so the Pi keeps the same address.
  • Rebooted a device (or toggled Wi-Fi) so it picks up the new setting.
Every router's menus are named differently. If you can't find the DNS field, search the web for "[your router model] change DNS DHCP". The setting always exists - it's just hidden in a different place.
Step 6

6. Check that it's working

Open the admin page (the address from Step 4) and log in with your password. Then browse the web for a minute and watch the numbers move.

  • The admin dashboard loads and you can log in.
  • "Total queries" is climbing - proof your devices are using the Pi.
  • "Queries blocked" is above zero - ads & trackers are being stopped.
  • Sites feel a little cleaner and faster. 🎉
  • Family check: on a device using the Pi, a Google image search for something mild-but-adult stays SafeSearch-clean, and a known adult site won't open.
Want a definite test? On a device using the Pi, visit http://pi.hole/admin - if it opens, your device is going through Pi-hole. Many ad-heavy news sites are the most satisfying "before/after".

If something looks off

A website or app won't load / a login is blocked

Very rarely a blocklist is too aggressive. Open the admin page -> Query Log, find the red (blocked) domain the site needs, and click Allow. It'll work within seconds - no reinstall needed.

Numbers aren't moving / devices aren't using Pi-hole

The router DNS change (Step 5) usually needs a device reboot or a Wi-Fi off/on to take effect. Also confirm the Pi still has the same IP you set in the router. On the Pi you can run pihole status to confirm it's active.

The Pi's address changed and nothing resolves

That's why the DHCP reservation in Step 5 matters. Find the Pi's new IP in the router, update the router's DNS setting to match, and make the reservation so it can't drift again.

Bonus

★ Set up another one

Once you've done it once, a second Pi-hole takes about ten minutes - handy for a second home, or a spare as backup. Two ways:

Fresh + same settings

Flash a new card, copy your edited setup.conf onto the new Pi, and run sudo ./bootstrap.sh. Identical result, every time.

Clone an existing one

On your working Pi run sudo ./backup.sh to save a Teleporter file, then on the new Pi restore it from Settings -> Teleporter -> Restore to copy your exact allow/block lists.

# on your working Pi - save a portable backup you can restore anywhere
$ sudo ./backup.sh
Keep them healthy: once a month, run sudo ./update.sh on each Pi to update the system, Pi-hole, and the blocklists.
